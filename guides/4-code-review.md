# 가이드 04: AI 코드 리뷰

> PR 리뷰를 Claude Code에게 맡기는 실전 워크플로

## 언제 쓰나요?

- PR을 올리기 전에 셀프 리뷰
- 팀원이 부재일 때 1차 리뷰
- 보안/성능 관점의 자동 체크

## PR 생성 전 (작성자)

### 셀프 리뷰

```bash
# 변경사항 전체 리뷰
claude "현재 브랜치의 변경사항을 리뷰해줘.
체크리스트:
- [ ] 보안 취약점
- [ ] 에러 핸들링 누락
- [ ] 타입 안전성
- [ ] 성능 이슈
- [ ] 테스트 커버리지"
```

### PR 설명 자동 생성

```bash
# PR 본문 초안 생성
claude "main 대비 변경사항을 분석하고 PR 설명을 작성해줘.
포맷:
## 변경 내용
## 왜 이렇게 했나
## 테스트 방법
## 스크린샷 (해당 시)"
```

## 방법 1: CLI에서 직접 리뷰

```bash
# 현재 브랜치의 변경사항 리뷰
claude "현재 브랜치의 변경사항을 리뷰해줘. 보안 취약점, 에러 핸들링 누락, 성능 이슈를 중심으로"

# 특정 파일만 리뷰
claude "src/api/auth.ts 파일을 리뷰해줘"
```

## 방법 2: 코드 리뷰 에이전트 활용

```bash
# .claude/agents/code-reviewer.md 생성
mkdir -p .claude/agents
```

```markdown
# .claude/agents/code-reviewer.md

## 역할
PR의 코드 변경사항을 리뷰합니다.

## 체크리스트
1. **보안** — 인젝션, XSS, 인증 우회 가능성
2. **에러 핸들링** — 누락된 try-catch, 빈 catch 블록
3. **타입 안전성** — any 사용, 타입 단언 남용
4. **성능** — N+1 쿼리, 불필요한 렌더링, 메모리 릭
5. **테스트** — 새 기능의 테스트 존재 여부

## 출력 형식
| 심각도 | 위치 | 이슈 | 제안 |
|--------|------|------|------|
| 🔴 Critical | file:line | ... | ... |
| 🟡 Warning | file:line | ... | ... |
| 🔵 Info | file:line | ... | ... |
```

## 방법 3: Git Diff 기반 리뷰

```bash
# main 대비 변경사항 리뷰
git diff main...HEAD | claude "이 diff를 리뷰해줘"

# 스테이징된 변경사항만
git diff --staged | claude "커밋 전 리뷰해줘"
```

## PR 리뷰 시 (리뷰어)

### 1차: 전체 파악

```bash
# PR의 변경사항 요약
gh pr diff [PR_NUMBER] | claude "이 PR의 변경사항을 요약해줘"
```

### 2차: 심층 리뷰

```bash
# 관점별 리뷰
gh pr diff [PR_NUMBER] | claude "다음 관점에서 리뷰해줘:
1. 이 변경이 기존 기능을 깨뜨리지 않는지
2. 엣지 케이스 처리
3. 성능 영향"
```

### 3차: 리뷰 코멘트 작성

```bash
claude "리뷰 결과를 GitHub PR 코멘트 형식으로 작성해줘.
- 반드시 고쳐야 할 것 (🔴)
- 권장 사항 (🟡)
- 칭찬할 점 (🟢)"
```

## 효과적인 리뷰 요청 패턴

**나쁜 예:**
```
코드 리뷰해줘
```

**좋은 예:**
```
이 PR은 사용자 인증 로직을 JWT에서 세션 기반으로 변경합니다.
다음 관점에서 리뷰해줘:
1. 세션 하이재킹 방지가 충분한지
2. 기존 API와의 하위 호환성
3. 로그아웃 시 세션 무효화 처리
```

## 리뷰 결과 활용

```bash
# 리뷰에서 발견된 이슈 바로 수정
claude "리뷰에서 지적된 N+1 쿼리 문제를 수정해줘"

# 수정 후 재리뷰
claude "수정한 부분을 다시 리뷰해줘"
```

## 리뷰 문화 팁

- AI 리뷰는 **1차 필터** — 최종 판단은 사람이
- "AI가 OK했으니 머지" ❌ → "AI가 놓친 것은?" ✅
- AI 리뷰 결과를 팀과 공유 → 리뷰 기준 통일 효과
