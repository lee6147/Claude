# 가이드 15: 서브에이전트 오케스트레이션

> 메인 에이전트가 서브에이전트를 생성·관리하며 복잡한 작업을 자동으로 분배하는 패턴

## 핵심 원칙

> **메인은 지휘자, 서브는 연주자 — 각자 맡은 파트만 완벽하게**

```
메인 에이전트 (Orchestrator)
  ├── 서브에이전트 A → 리서치
  ├── 서브에이전트 B → 코드 생성
  └── 서브에이전트 C → 리뷰 & 검증
```

## 에이전트 팀 vs 서브에이전트

| 에이전트 팀 (가이드 11) | 서브에이전트 오케스트레이션 |
|------------------------|--------------------------|
| 수동 프롬프트 분배 | 메인이 자동으로 태스크 할당 |
| 동시 시작, 수동 통합 | 의존성 기반 순차/병렬 자동 |
| tmux 세션으로 관리 | API/프레임워크로 관리 |
| 사람이 결과 병합 | 메인 에이전트가 결과 수집·병합 |

핵심 차이는 **자동화 수준**이에요. 서브에이전트 패턴에서는 메인 에이전트가 작업 분배부터 결과 수집까지 직접 해요.

## Step 1: 작업 분해 설계

서브에이전트에게 넘길 작업을 설계할 때 3가지 기준으로 판단해요.

### 서브에이전트로 분리하면 좋은 경우

| 기준 | 설명 | 예시 |
|------|------|------|
| **독립성** | 다른 작업 결과 없이 시작 가능 | 웹 검색, 파일 읽기 |
| **시간 소요** | 10초 이상 걸리는 작업 | API 크롤링, 대용량 분석 |
| **전문성** | 특화된 도구/컨텍스트 필요 | 브라우저 자동화, DB 쿼리 |

### 서브에이전트로 분리하면 안 되는 경우

- 사용자와 대화가 필요한 작업 (서브에이전트는 사용자와 직접 소통 못 함)
- 메인 컨텍스트에 강하게 의존하는 작업
- 1-2초면 끝나는 단순 작업

## Step 2: 서브에이전트 프롬프트 작성

좋은 서브에이전트 프롬프트의 구조:

```markdown
## 역할
당신은 [구체적 역할]입니다.

## 태스크
[정확히 무엇을 해야 하는지]

## 입력
- [메인에이전트가 전달하는 데이터]

## 출력 형식
- [정확한 출력 포맷 지정]

## 제약 조건
- [하면 안 되는 것 명시]
```

### 좋은 예시 vs 나쁜 예시

```markdown
# ❌ 나쁜 프롬프트
"AI 뉴스를 찾아줘"

# ✅ 좋은 프롬프트
"2026년 3월 둘째 주 AI 코딩 도구 관련 뉴스를 5개 찾아줘.
각 뉴스는 {title, url, summary(2줄), relevance_score(1-10)} 형식으로.
출처는 반드시 영문 테크 미디어만 사용."
```

## Step 3: 오케스트레이션 패턴

### 패턴 1: Fan-Out / Fan-In

가장 흔한 패턴. 여러 서브에이전트에게 동시에 작업을 주고 결과를 모아요.

```
메인 ──┬── 서브A (뉴스 수집) ──┐
       ├── 서브B (트렌드 분석) ──┼── 메인 (종합)
       └── 서브C (경쟁사 분석) ──┘
```

**적합한 경우:** 리서치, 다각도 분석, 벤치마크 비교

### 패턴 2: Pipeline (직렬 체이닝)

앞 단계 결과가 다음 단계 입력이 되는 패턴.

```
서브A (데이터 수집) → 서브B (분석) → 서브C (보고서 작성)
```

**적합한 경우:** ETL, 콘텐츠 파이프라인, 코드 생성→리뷰→수정

### 패턴 3: Supervisor

메인이 서브에이전트 결과를 평가하고 재작업을 지시하는 패턴.

```
메인 → 서브A (초안 작성)
메인 ← 서브A (초안 반환)
메인: "품질 부족, 이 부분 수정해"
메인 → 서브A (수정 작업)
메인 ← 서브A (최종본)
```

**적합한 경우:** 품질이 중요한 콘텐츠, 코드 리뷰 반복

## Step 4: 에러 핸들링

서브에이전트가 실패할 수 있어요. 대비 전략:

| 상황 | 대응 |
|------|------|
| 타임아웃 | 재시도 1회 후 메인이 직접 처리 |
| 잘못된 출력 형식 | 출력 검증 후 재요청 |
| 부분 실패 | 성공한 결과만 사용, 실패 부분 로그 |
| 전체 실패 | 메인이 fallback 로직 실행 |

## Step 5: 실전 예시 — 주간 뉴스레터 자동화

```
메인 에이전트: "이번 주 AI 뉴스레터를 만들어줘"

1. Fan-Out 단계
   ├── 서브A: AI 코딩 도구 뉴스 수집 (5개)
   ├── 서브B: LLM 업데이트 뉴스 수집 (5개)
   └── 서브C: 개발 생산성 뉴스 수집 (5개)

2. Pipeline 단계
   서브D: 15개 뉴스 → 중복 제거 → 중요도 순 정렬 → Top 7 선정

3. Supervisor 단계
   서브E: Top 7으로 뉴스레터 초안 작성
   메인: 초안 검토 → 피드백 → 서브E 수정 → 최종본
```

## 핵심 체크리스트

- [ ] 서브에이전트 태스크가 독립적으로 실행 가능한가?
- [ ] 프롬프트에 입력/출력 형식이 명확한가?
- [ ] 타임아웃과 에러 핸들링이 있는가?
- [ ] 서브에이전트 수가 적절한가? (보통 3-5개가 최적)
- [ ] 결과 병합 로직이 메인에 있는가?
