# 가이드 06: 리팩토링 워크플로

> 기존 코드를 AI와 함께 안전하게 개선하는 단계별 프로세스

## 핵심 원칙

> **리팩토링의 황금률:** 동작을 바꾸지 않으면서 구조를 개선한다.

Claude Code로 리팩토링할 때 가장 흔한 실수:
- ❌ "이 파일 전체를 리팩토링해줘" → 범위가 너무 넓어 버그 유입
- ✅ "이 함수를 추출하고 테스트가 통과하는지 확인해줘" → 안전한 단위

## Step 1: 현황 파악

```bash
# Claude에게 현재 코드 분석 요청
claude "src/services/payment.ts를 분석해줘.
- 함수별 복잡도
- 중복 코드
- 개선 가능한 패턴
수정하지 말고 분석만 해줘."
```

## Step 2: 리팩토링 계획

```bash
claude "분석 결과를 바탕으로 리팩토링 계획을 세워줘.
조건:
1. 각 단계는 독립적으로 커밋 가능해야 함
2. 기존 테스트가 항상 통과해야 함
3. 한 단계에서 3개 이상 파일을 수정하지 않을 것"
```

**좋은 계획 예시:**

| 단계 | 변경 | 영향 범위 |
|------|------|----------|
| 1 | 유틸 함수 추출 | 1 파일 |
| 2 | 타입 정의 분리 | 2 파일 |
| 3 | 에러 핸들링 통합 | 2 파일 |
| 4 | 테스트 보강 | 1 파일 |

## Step 3: 단계별 실행

```bash
# 각 단계마다 테스트 확인
claude "리팩토링 1단계를 실행해줘. 완료 후 테스트 실행해서 결과 알려줘."

# 테스트 통과 확인 후 커밋
git add -A && git commit -m "refactor: extract utility functions"

# 다음 단계
claude "2단계 진행해줘."
```

## 안전장치

### 테스트가 없는 코드 리팩토링

```bash
# Step 0: 먼저 테스트 추가
claude "src/services/payment.ts의 주요 함수에 대해
현재 동작을 보존하는 테스트를 먼저 작성해줘.
리팩토링은 하지 말고 테스트만."

# 테스트 통과 확인
npm test

# 이제 리팩토링 시작
claude "테스트가 통과하는 상태에서 리팩토링 시작해줘."
```

### 롤백 포인트

```bash
# 리팩토링 시작 전 태그
git tag before-refactor

# 문제 발생 시 롤백
git reset --hard before-refactor
```

## 리팩토링 패턴 (자주 쓰는 것)

### 1. 함수 추출

```bash
claude "processPayment 함수에서 검증 로직을 validatePayment로 추출해줘"
```

### 2. 타입 강화

```bash
claude "이 파일에서 any 타입을 모두 구체적인 타입으로 교체해줘"
```

### 3. 중복 제거

```bash
claude "src/api/users.ts와 src/api/posts.ts에서 중복되는
에러 핸들링 로직을 공통 미들웨어로 추출해줘"
```

### 4. 패턴 적용

```bash
claude "이 switch문을 Strategy 패턴으로 리팩토링해줘.
기존 테스트가 통과해야 해."
```

## 체크리스트

- [ ] 리팩토링 전 테스트 커버리지 확인 (없으면 먼저 추가)
- [ ] 계획 수립 → 단계 분리
- [ ] 각 단계마다 테스트 통과 확인
- [ ] 각 단계마다 커밋
- [ ] 전체 완료 후 통합 테스트

## 주의사항

- **한 번에 너무 많이 바꾸지 않기** — 단계당 최대 3파일
- **"더 좋은 코드"의 유혹 저항** — 계획에 없는 개선은 별도 PR
- **기존 API/인터페이스 유지** — 외부 계약을 바꾸면 리팩토링이 아님
