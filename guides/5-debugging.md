# 가이드 05: AI 디버깅 워크플로

> 에러를 만났을 때 Claude Code로 체계적으로 해결하는 프로세스

## 언제 쓰나요?

- 런타임 에러가 발생했을 때
- 빌드/배포 실패 원인을 찾을 때
- 간헐적 버그를 추적할 때
- 성능 병목을 분석할 때

## Step 1: 에러 컨텍스트 수집

디버깅 효율은 **얼마나 좋은 컨텍스트를 제공하느냐**에 달렸습니다.

```bash
# 에러 로그를 파이프로 전달
npm run build 2>&1 | claude "이 빌드 에러를 분석해줘"

# 특정 에러 메시지와 함께
claude "다음 에러가 발생합니다:
에러: TypeError: Cannot read property 'id' of undefined
파일: src/api/users.ts:42
재현: GET /api/users/999 (존재하지 않는 ID)
예상 동작: 404 반환
실제 동작: 500 에러"
```

## Step 2: 가설 수립

```bash
claude "이 에러의 가능한 원인 3가지를 우선순위로 나열해줘.
각 원인에 대해:
1. 왜 그렇게 생각하는지
2. 어떻게 확인할 수 있는지
3. 수정 방법
코드를 수정하지 말고 분석만 해줘."
```

**좋은 응답 예시:**

| 순위 | 가설 | 확인 방법 |
|------|------|----------|
| 1 | DB 쿼리 결과 null 체크 누락 | users.ts:40 확인 |
| 2 | 미들웨어에서 파라미터 파싱 실패 | req.params 로깅 |
| 3 | 캐시에서 만료된 데이터 반환 | Redis TTL 확인 |

## Step 3: 체계적 수정

```bash
# 가설 1번부터 확인 및 수정
claude "가설 1번 (null 체크 누락)을 확인하고,
맞다면 수정해줘. 에러 케이스에 대한 테스트도 추가해줘."

# 수정 후 재현 테스트
npm test
```

## 디버깅 패턴

### 패턴 1: 스택 트레이스 분석

```bash
# 전체 스택 트레이스 전달
claude "이 스택 트레이스를 분석해줘:
[스택 트레이스 붙여넣기]

관련 소스 코드도 함께 확인해서 root cause를 찾아줘."
```

### 패턴 2: 로그 기반 추적

```bash
# 로그 파일 분석
claude "logs/error.log의 최근 100줄을 분석해줘.
반복되는 패턴이나 에러 원인을 찾아줘."

# 디버그 로그 추가 요청
claude "이 함수의 입출력에 디버그 로그를 추가해줘.
프로덕션에서 제거하기 쉽도록 DEBUG 플래그로 감싸줘."
```

### 패턴 3: 간헐적 버그

```bash
claude "이 버그는 10번 중 1번 정도 발생합니다.
증상: [설명]
의심 영역: [파일/함수]

Race condition, 타이밍 이슈, 캐시 관련 가능성을 중심으로
분석해줘."
```

### 패턴 4: 환경별 차이

```bash
claude "로컬에서는 동작하지만 프로덕션에서 실패합니다.
- 로컬: Node 20, macOS, SQLite
- 프로덕션: Node 20, Linux, PostgreSQL

환경 차이에서 올 수 있는 원인을 분석해줘."
```

## 흔한 실수

| 실수 | 개선 |
|------|------|
| "에러 고쳐줘" | 에러 메시지, 재현 조건, 예상 동작 포함 |
| 스택 트레이스 생략 | 전체 스택 트레이스 복사 |
| 바로 수정 요청 | 분석 먼저 → 확인 → 수정 |
| 하나만 수정하고 끝 | 같은 패턴의 다른 위치도 확인 |

## 디버깅 체크리스트

- [ ] 에러 메시지 전문 확보
- [ ] 재현 조건 정리
- [ ] 가설 수립 (최소 2개)
- [ ] 가설 검증 (분석 먼저, 수정은 나중에)
- [ ] 수정 후 테스트 통과 확인
- [ ] 같은 패턴의 다른 위치 확인
- [ ] 재발 방지 (테스트 추가, CLAUDE.md에 주의사항 기록)
