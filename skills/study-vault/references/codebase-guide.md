# 코드베이스 분석 가이드

> 코드베이스를 학습 자료로 변환할 때 참조하는 워크플로우입니다.
> study-vault의 확장 모드로, 소스 코드를 Obsidian 노트로 정리합니다.

---

## 언제 사용하나요?

사용자가 다음과 같이 요청할 때:

- "이 프로젝트 구조를 정리해줘"
- "코드베이스를 학습 노트로 만들어줘"
- 오픈소스 프로젝트를 분석하고 싶을 때

## 분석 단계

### 1단계: 프로젝트 스캔

```
- [ ] 루트 디렉토리 구조 파악
- [ ] README, 설정 파일(package.json, pyproject.toml 등) 읽기
- [ ] 기술 스택 식별
- [ ] 진입점(entry point) 찾기
```

| 확인 항목 | 방법 |
|----------|------|
| 언어/프레임워크 | 설정 파일, 확장자 분포 |
| 아키텍처 패턴 | 디렉토리 구조, import 관계 |
| 핵심 모듈 | 가장 많이 import되는 파일 |
| 테스트 구조 | test/, __tests__/, *_test.* |

### 2단계: 모듈 분류

프로젝트를 **학습 가능한 단위**로 분류합니다:

| 분류 | 기준 | 예시 |
|------|------|------|
| **핵심 모듈** | 비즈니스 로직 | services/, domain/ |
| **인프라 모듈** | 외부 연결 | db/, api/, config/ |
| **UI 모듈** | 화면/인터페이스 | components/, pages/ |
| **유틸리티** | 공통 도구 | utils/, helpers/ |

```
- [ ] 각 모듈별 파일 목록 작성
- [ ] 모듈 간 의존 관계 파악
- [ ] 학습 순서 결정 (의존성 기반)
```

### 3단계: 토픽 매핑

각 모듈을 StudyVault 토픽으로 변환합니다:

```
모듈 → 토픽 폴더
├── 개념노트.md  ← 모듈의 역할, 주요 클래스/함수, 설계 의도
└── 연습문제.md  ← 코드 이해도 퀴즈
```

```
- [ ] 모듈당 개념노트 작성 (핵심 함수/클래스 설명)
- [ ] 코드 스니펫 포함 (10줄 이내)
- [ ] 아키텍처 결정 이유(ADR) 포함 (있는 경우)
- [ ] 모듈당 연습문제 5개 이상
```

### 4단계: 아키텍처 노트

프로젝트 전체 구조를 Dashboard에 정리합니다:

```
- [ ] 전체 아키텍처 다이어그램 (텍스트)
- [ ] 데이터 흐름 설명
- [ ] 주요 설계 결정 목록
- [ ] 실행/빌드 명령어 정리
```

## 연습문제 유형 (코드베이스용)

코드베이스 학습에 적합한 문제 유형:

| 유형 | 설명 | 비율 |
|------|------|------|
| **코드 읽기** | "이 함수의 반환값은?" | 30% |
| **설계 이해** | "이 패턴을 사용한 이유는?" | 30% |
| **의존성** | "A 모듈이 변경되면 영향받는 것은?" | 20% |
| **실습** | "이 기능을 추가하려면 어디를 수정?" | 20% |

## 주의 사항

- 코드 전체를 복사하지 않음 — 핵심 스니펫(10줄 이내)만 인용
- `.env`, 시크릿, 개인정보가 포함된 파일은 건너뜀
- 너무 큰 프로젝트(100+ 파일)는 핵심 모듈만 선별
