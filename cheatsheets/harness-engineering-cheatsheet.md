# 하네스 엔지니어링 치트시트

> Model / Harness / Surfaces 구조 한눈에 보기 — 체크리스트, 패턴, 안티패턴 원페이저

## 핵심 개념: 왜 하네스인가?

| 구분 | 프롬프트 엔지니어링 | 하네스 엔지니어링 |
|------|---------------------|-------------------|
| 초점 | 모델 입력 최적화 | 실행 환경 전체 설계 |
| 범위 | 단일 턴 품질 향상 | 멀티턴 워크플로우 안정성 |
| 성과 기여 | ~20% | ~80% |
| 비유 | 운전자에게 지시 | 도로 + 신호 + 가드레일 설계 |

> 모델 성능의 병목은 프롬프트가 아니라 하네스에 있다.

## 3-Layer 아키텍처

```
┌─────────────────────────────┐
│  Surfaces (사용자 인터페이스) │  ← IDE, CLI, 웹 UI, API
├─────────────────────────────┤
│  Harness (실행 환경)         │  ← 도구, 권한, 컨텍스트, 루프
├─────────────────────────────┤
│  Model (AI 모델)            │  ← Claude, GPT, Gemini 등
└─────────────────────────────┘
```

| 레이어 | 역할 | 예시 |
|--------|------|------|
| **Model** | 추론, 코드 생성 | Claude Opus, GPT-5 |
| **Harness** | 도구 제공, 권한 제어, 반복 루프 | Claude Code, Codex CLI, Cursor |
| **Surfaces** | 사용자와의 접점 | 터미널, IDE 패널, 슬랙봇 |

## 하네스 설계 체크리스트

### 도구(Tools)

- [ ] 파일 읽기/쓰기 도구 제공
- [ ] 터미널 명령 실행 (샌드박스 여부 결정)
- [ ] 웹 검색/브라우징 도구
- [ ] 테스트 실행기 연결
- [ ] 린터/포매터 자동 실행

### 컨텍스트 관리

- [ ] 프로젝트 구조 자동 주입 (CLAUDE.md, .cursorrules)
- [ ] 관련 파일만 선택적 로딩
- [ ] 이전 세션 요약 전달 (Session Summary Bridge)
- [ ] 토큰 사용량 모니터링

### 반복 루프(Iteration Loop)

- [ ] 코드 생성 → 테스트 → 수정 자동 사이클
- [ ] 에러 시 자동 재시도 (최대 N회)
- [ ] 린트 에러 자동 피드백
- [ ] diff 기반 변경 검증

### 안전장치(Guardrails)

- [ ] 파일 수정 전 확인 프롬프트
- [ ] 위험 명령어 차단 목록
- [ ] 비용/토큰 제한 설정
- [ ] 타임아웃 설정

## 실전 패턴 5가지

### 1. 테스트 루프 패턴

```yaml
# 가장 효과적인 하네스 패턴
loop:
  1. 코드 생성
  2. 테스트 실행 (자동)
  3. 실패 시 → 에러 메시지 피드백
  4. 수정 후 재실행
  5. 전체 통과 시 종료
```

테스트가 있는 환경에서 에이전트 성공률이 **2~3배** 높아진다.

### 2. CLAUDE.md 패턴 (프로젝트 하네스)

```markdown
# CLAUDE.md — 프로젝트 루트에 배치

## 프로젝트 개요
{한 줄 설명}

## 아키텍처
{디렉토리 구조, 주요 모듈}

## 규칙
- 테스트 없이 커밋 금지
- TypeScript strict 모드 필수
```

에이전트가 매 세션 시작 시 자동으로 읽어서 컨텍스트를 확보한다.

### 3. 도구 제한 패턴

```json
{
  "permissions": {
    "file_write": ["src/**", "tests/**"],
    "file_read": ["**"],
    "exec": ["npm test", "npm run lint"],
    "web": false
  }
}
```

필요한 도구만 열어두면 에이전트가 엉뚱한 작업을 하지 않는다.

### 4. 멀티 에이전트 오케스트레이션

```
Orchestrator (계획 수립)
  ├── Agent A: 프론트엔드 구현
  ├── Agent B: 백엔드 API
  └── Agent C: 테스트 작성

각 에이전트는 독립 하네스(별도 워킹 디렉토리, 도구셋)에서 실행
```

| 항목 | 단일 에이전트 | 멀티 에이전트 |
|------|--------------|--------------|
| 컨텍스트 충돌 | 빈번 | 없음 |
| 병렬 처리 | 불가 | 가능 |
| 디버깅 | 쉬움 | 로그 분리 필요 |

### 5. Hooks 패턴 (이벤트 기반 하네스)

```json
{
  "hooks": {
    "PreToolUse": {
      "Bash": "echo '위험 명령어 체크'",
      "Write": "echo '파일 백업 생성'"
    },
    "PostToolUse": {
      "Write": "npm run lint --fix"
    }
  }
}
```

도구 실행 전후에 자동으로 검증/보정 로직을 삽입한다.

## 안티패턴 (이렇게 하면 실패한다)

| 안티패턴 | 문제점 | 개선 방향 |
|----------|--------|-----------|
| 프롬프트만 개선 | 환경이 나쁘면 소용없음 | 하네스 먼저 정비 |
| 모든 파일 컨텍스트에 넣기 | 토큰 낭비, 정확도 하락 | 관련 파일만 선택적 로딩 |
| 테스트 없이 에이전트 실행 | 검증 불가, 버그 누적 | 테스트 루프 필수 |
| 무제한 도구 권한 | 시스템 파일 삭제 위험 | 최소 권한 원칙 |
| 에러 무시하고 계속 실행 | 에러 위에 에러 쌓임 | 에러 시 즉시 중단 + 피드백 |
| 하나의 거대한 프롬프트 | 맥락 혼란 | 태스크 분할 + 각각 독립 실행 |

## 도구별 하네스 비교

| 기능 | Claude Code | Codex CLI | Cursor | Windsurf |
|------|-------------|-----------|--------|----------|
| CLAUDE.md / 규칙 파일 | ✅ CLAUDE.md | ✅ AGENTS.md | ✅ .cursorrules | ✅ .windsurfrules |
| 테스트 자동 실행 | ✅ 내장 | ✅ 내장 | ⚠️ 수동 | ⚠️ 수동 |
| 파일 권한 제어 | ✅ 허용 목록 | ✅ 샌드박스 | ❌ | ❌ |
| 멀티 에이전트 | ✅ subagent | ✅ 멀티 세션 | ⚠️ 제한적 | ❌ |
| Hooks / 이벤트 | ✅ Pre/Post | ❌ | ❌ | ❌ |
| MCP 도구 연결 | ✅ | ❌ | ✅ | ✅ |

## 하네스 성숙도 단계

```
Level 0: 프롬프트만 사용 (채팅창에 복붙)
Level 1: 프로젝트 규칙 파일 작성 (CLAUDE.md)
Level 2: 도구 + 권한 설정 (테스트 루프 포함)
Level 3: Hooks + 자동화 파이프라인
Level 4: 멀티 에이전트 오케스트레이션 + 감독 체계
```

대부분의 팀은 Level 1에 머물러 있다. Level 2만 달성해도 생산성이 크게 올라간다.

## 빠른 시작: 5분 하네스 구축

```bash
# 1. 프로젝트 루트에 CLAUDE.md 생성
cat > CLAUDE.md << 'EOF'
# 프로젝트 개요
{한 줄 설명}

# 규칙
- 테스트 통과 확인 후 커밋
- TypeScript strict 모드
EOF

# 2. 테스트 명령어 확인
npm test  # 또는 pytest, go test

# 3. Claude Code 실행 — 하네스가 자동으로 적용됨
claude
```

이것만으로 Level 1 하네스가 완성된다.
