# 에이전틱 코딩 패턴 치트시트

> 멀티 에이전트 협업으로 복잡한 개발을 체계적으로 처리하는 핵심 패턴 — 한 페이지 요약

## 에이전틱 코딩이란?

AI 에이전트에게 단순 코드 생성이 아니라 **태스크 분할, 실행, 검증**까지 위임하는 개발 방식이에요. 2026년에는 멀티 에이전트 협업이 실전에서 본격적으로 쓰이기 시작했어요.

## 핵심 패턴 5가지

| 패턴 | 설명 | 사용 시점 |
|------|------|----------|
| Orchestrator-Worker | 메인 에이전트가 태스크를 분할하고 서브 에이전트에게 위임 | 대규모 리팩토링, 멀티파일 변경 |
| Parallel Fan-out | 독립적인 태스크를 여러 에이전트가 동시 처리 | 테스트 생성, 문서화, 린트 |
| Pipeline (Chain) | 에이전트 A → B → C 순서로 결과를 넘기며 처리 | 코드 생성 → 리뷰 → 테스트 |
| Specialist Router | 요청 유형에 따라 전문 에이전트로 라우팅 | 프론트/백/인프라 분리 작업 |
| Verify-Loop | 생성 → 검증 → 수정을 반복하여 품질 확보 | 버그 수정, 보안 패치 |

## Orchestrator-Worker 패턴

가장 많이 쓰이는 패턴이에요. 하나의 "감독자" 에이전트가 큰 태스크를 쪼개서 여러 워커에게 넘겨요.

```bash
# Claude Code에서 멀티 에이전트 실행
claude "이 프로젝트의 모든 API 엔드포인트에 입력 검증을 추가해줘.
각 파일별로 서브태스크를 나눠서 병렬로 처리해."

# tmux로 직접 병렬 실행
tmux new-session -d -s agent1 'claude -p "src/api/users.ts에 입력 검증 추가"'
tmux new-session -d -s agent2 'claude -p "src/api/posts.ts에 입력 검증 추가"'
tmux new-session -d -s agent3 'claude -p "src/api/auth.ts에 입력 검증 추가"'
```

### 태스크 분할 기준

| 기준 | 좋은 분할 | 나쁜 분할 |
|------|----------|----------|
| 파일 의존성 | 독립된 파일 각각 처리 | 서로 참조하는 파일을 분리 |
| 변경 범위 | 하나의 관심사만 다룸 | 여러 레이어를 한꺼번에 |
| 검증 가능성 | 개별 테스트로 확인 가능 | 전체를 합쳐야 확인 가능 |

## Parallel Fan-out 패턴

독립적인 작업을 동시에 돌려서 시간을 줄여요.

```bash
# 3개 에이전트가 동시에 각각 다른 작업 수행
# Agent 1: 테스트 생성
claude -p "src/services/ 하위 모든 파일의 유닛 테스트 생성" &

# Agent 2: 문서 생성
claude -p "src/services/ 하위 모든 파일의 JSDoc 추가" &

# Agent 3: 타입 강화
claude -p "src/services/ 하위 any 타입을 구체적 타입으로 교체" &

wait  # 모든 에이전트 완료 대기
```

## Pipeline 패턴

단계별로 결과를 넘기면서 품질을 쌓아가요.

```bash
# Stage 1: 코드 생성
claude -p "사용자 인증 API를 구현해줘" > /tmp/stage1.txt

# Stage 2: 코드 리뷰
cat /tmp/stage1.txt | claude -p "이 코드를 리뷰해줘. 보안, 에러핸들링 체크" > /tmp/stage2.txt

# Stage 3: 리뷰 반영
cat /tmp/stage2.txt | claude -p "리뷰 피드백을 반영해서 코드를 수정해줘"
```

## 컨텍스트 전달 패턴

멀티 에이전트에서 가장 중요한 건 **컨텍스트를 어떻게 넘기느냐**예요.

| 방법 | 장점 | 단점 |
|------|------|------|
| 파일 기반 | 안정적, 추적 가능 | 동기화 필요 |
| 프롬프트 주입 | 간단, 즉시 적용 | 토큰 비용 증가 |
| CLAUDE.md | 모든 에이전트가 자동 로드 | 프로젝트 전체에 영향 |
| 환경변수 | 설정값 전달에 적합 | 복잡한 데이터 부적합 |

### CLAUDE.md로 공유 컨텍스트 설정

```markdown
# CLAUDE.md

## 프로젝트 규칙
- TypeScript strict 모드
- 에러는 커스텀 AppError 클래스 사용
- 모든 public 함수에 JSDoc 필수

## 현재 작업
- [ ] API 입력 검증 추가 (agent-1 담당)
- [ ] 테스트 커버리지 80% 달성 (agent-2 담당)
- [ ] README 업데이트 (agent-3 담당)
```

## 흔한 실수 & 해결

| 실수 | 해결 |
|------|------|
| 에이전트끼리 같은 파일 수정 → 충돌 | 파일 단위로 작업 범위를 명확히 분리 |
| 컨텍스트 부족으로 일관성 깨짐 | CLAUDE.md에 공통 규칙 정의 |
| 너무 작게 쪼개서 오버헤드 증가 | 최소 단위: 하나의 파일 또는 하나의 기능 |
| 검증 없이 결과를 그대로 머지 | Pipeline 끝에 반드시 Verify 단계 추가 |
| 병렬 에이전트 수가 너무 많음 | API 호출 한도 고려, 3~5개가 적정 |

## 에이전틱 코딩 체크리스트

- [ ] 태스크가 독립적으로 분할 가능한가?
- [ ] 각 에이전트의 작업 범위가 명확한가?
- [ ] 공통 규칙(CLAUDE.md)이 정의되어 있는가?
- [ ] 결과 검증 단계가 포함되어 있는가?
- [ ] 충돌 가능성을 사전에 차단했는가?
