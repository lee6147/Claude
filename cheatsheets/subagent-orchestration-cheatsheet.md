# 서브에이전트 오케스트레이션 치트시트

> 서브에이전트 패턴을 빠르게 적용하기 위한 레퍼런스 카드

## 언제 서브에이전트를 쓸까?

| ✅ 사용 | ❌ 비사용 |
|---------|----------|
| 10초+ 걸리는 독립 작업 | 1-2초 단순 작업 |
| 병렬 가능한 리서치 | 사용자 대화 필요 |
| 전문 도구 필요 (브라우저 등) | 메인 컨텍스트 의존적 |

## 3대 패턴 요약

```
Fan-Out/In:   메인 ──┬── A ──┐── 메인 (병렬 수집)
                     ├── B ──┤
                     └── C ──┘

Pipeline:     A → B → C (직렬 체이닝)

Supervisor:   메인 → A → 메인(검토) → A(수정) → 메인(승인)
```

## 프롬프트 템플릿

```markdown
## 역할: [한 줄]
## 태스크: [구체적 지시]
## 입력: [데이터/파라미터]
## 출력 형식: [JSON/마크다운/등]
## 제약: [금지 사항]
```

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| 타임아웃 | 1회 재시도 → fallback |
| 형식 오류 | 검증 후 재요청 |
| 부분 실패 | 성공분만 사용 |

## 최적 서브에이전트 수

- **리서치**: 3-5개 (주제별 분배)
- **코드 생성**: 2-3개 (모듈별 분배)
- **리뷰**: 1-2개 (관점별 분배)

> 💡 서브에이전트가 5개 넘으면 메인의 조율 비용이 급증해요

## 결과 병합 팁

1. 각 서브에이전트 출력을 **동일 JSON 스키마**로 통일
2. 메인에서 `dedup → rank → filter → merge` 순서로 처리
3. 최종 결과에 출처(어떤 서브에이전트)를 표기
